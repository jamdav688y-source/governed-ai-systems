# Escalation Logic Runbook

## Purpose

Escalation means increasing precision, not aggression.

The objective is to move the target from abstraction to explicit system recognition.

---

## When Escalation Is Allowed

Escalate only when:
- signal is still present
- a real structural gap is visible
- the next move can be narrower than the previous one

Do not escalate for its own sake.

---

## Escalation Triggers

### 1. Partial Certainty
Examples:
- "we think"
- "this should"
- "probably"
- "it seems"

### 2. Defensive Confidence
Examples:
- "we handle that with monitoring"
- "our safeguards cover this"
- "we already account for that"

### 3. Constraint Acknowledgement
Examples:
- "that's the hard part"
- "that's the real issue"
- "this is where it breaks"

### 4. System-Localized Uncertainty
Examples:
- "not sure how to enforce that"
- "under scale it changes"
- "we lose the boundary there"

---

## Escalation by Target Type

### Type A
Goal:
- localize to one execution path
- expose authority/evaluation mismatch
- move toward conversion readiness

Avoid:
- broad theory
- long conceptual detours

### Type B
Goal:
- add one missing invariant
- sharpen one hidden failure mode
- deepen alignment

Avoid:
- selling
- diagnostic framing without ownership

### Type C
Goal:
- structure the discomfort
- turn vague tension into a named failure condition
- introduce one concrete mechanism

Avoid:
- premature conversion
- multiple abstractions at once

### Type D/E
Goal:
- conserve energy
- filter or drop

---

## Canonical Escalation Patterns

### Pattern 1 — Missing Invariant
Use when target is structurally strong but incomplete.

Shape:
- validate their model
- add one deeper invariant
- stop

### Pattern 2 — Boundary Compression
Use when target is spread across abstractions.

Shape:
- isolate one transition
- define one failure
- bind it to one control requirement

### Pattern 3 — Ownership Activation
Use when target is speaking conceptually but hints at real exposure.

Shape:
- ask where it fails in their environment
- localize to one implementation seam
- wait

---

## De-Escalation Rules

De-escalate when:
- signal drops
- target becomes generic
- thread turns performative
- resistance repeats without new structure
- topic drifts

Valid de-escalation outcomes:
- HOLD
- EXIT
- archive for later re-entry

---

## Logging

Every escalation must record:
- prior stage
- escalation trigger
- escalation pattern used
- message sent
- response type received
- whether signal increased, stayed flat, or decreased
